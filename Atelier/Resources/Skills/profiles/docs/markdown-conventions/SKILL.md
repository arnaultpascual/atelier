# Markdown Conventions

## Frontmatter
- YAML at top, between `---` fences, no trailing whitespace.
- Required fields depend on the site generator. Check existing docs first.
- Common: `title`, `description`, `date`, `tags`, `slug`.

```yaml
---
title: My doc
description: One-line summary, shows in search.
---
```

## Heading hierarchy
- Exactly one `#` (h1) per document — usually matches `title` in frontmatter.
- Don't skip levels: `##` after `#`, then `###`, etc.
- Sentence case for headings unless project uses title case (check existing).

## Code blocks
- Always tag the language for syntax highlighting: ``` ```bash ```
- Inline code with backticks for filenames, commands, code identifiers.
- For shell examples: prompt-free, copy-pasteable. NOT `$ command`.

## Links
- Relative for in-repo: `[foo](../guide/foo.md)`.
- Absolute URL for external. Prefer HTTPS.
- Don't use bare URLs in prose — wrap as proper links.
- Anchor links: `[Section](#section)` — generated by lowercasing the heading, replacing spaces with `-`.

## Don't
- Mix `*` and `_` for emphasis in the same file. Pick one.
- Use `<br>` to force a line break. Use a blank line for a paragraph break.
- Leave TODO / FIXME in shipped docs without an issue link.
- Auto-generate ToCs if the site generator does it for you.

## Verify
- Renders correctly in the project's docs preview (mkdocs, docusaurus, vitepress, etc.).
- Spell check via `cspell` or built-in tool if the project has one configured.
- Links not dead: `lychee` or similar if available.
